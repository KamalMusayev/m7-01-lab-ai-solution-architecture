# Justification — Weekly Churn Prediction System

I chose **batch inference** for this system because the business requirement does not need real-time predictions. In Scenario B, the sales team wants “a ranked list of accounts at risk of churning, delivered every Monday morning.” The scenario also says that the team will revisit the list throughout the week, but “the underlying score only needs to refresh weekly.” These two details make batch serving the best fit. No user is waiting on an immediate prediction, and there is no live event that needs to be scored in milliseconds. Instead, the system can compute churn scores on a fixed weekly schedule and write the results to a table used by the CRM dashboard and outreach playbooks.

Inference should run in the **cloud**. This is a B2B SaaS churn prediction problem, so the input data is likely already stored in cloud systems such as a data warehouse, CRM, billing database, and product analytics platform. Running inference at the edge would not make sense because there is no device-level prediction need and no strict user-facing latency requirement. Cloud batch inference is easier to schedule, monitor, retry, and scale. It also allows the system to process all 120,000 accounts in one controlled job instead of maintaining an always-on online prediction API.

The two main targets I would optimize are **cost** and **reliability/throughput**. Cost matters because the model only needs to refresh weekly, so keeping an online service running 24/7 would waste infrastructure. A scheduled batch job can use compute only when needed, then shut down after scoring finishes. Throughput also matters because the system must score around 120,000 accounts before the Monday morning sales workflow begins. The latency budget is not measured in milliseconds here; the practical budget is that the full scoring job must complete before Monday 8:00 AM, when the sales team starts using the ranked churn list. As long as the batch job finishes before that deadline, sub-second prediction latency is not important.

The fallback plan is to use the **last successful churn score table** if the weekly model run fails. The CRM dashboard should clearly mark those scores as stale, for example “last updated 7 days ago,” so the sales team understands that the list is not freshly scored. If the model is available but monitoring shows poor quality, such as unusual score drift or a drop in recent validation performance, the system should fall back to simple rule-based risk segments. Examples could include accounts with falling product usage, unpaid invoices, or upcoming renewal dates. These rules are less powerful than the model, but they are explainable and safer than showing unreliable predictions.

This design accepts some prediction staleness in exchange for lower cost, simpler operations, and a more reliable weekly workflow. Since the business process itself is weekly, batch inference matches the way the sales team actually consumes the predictions.
